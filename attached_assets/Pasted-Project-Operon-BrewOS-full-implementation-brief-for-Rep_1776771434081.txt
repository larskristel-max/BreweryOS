Project: Operon / BrewOS — full implementation brief for Replit

Important:
There is also an `agent.md` file in the GitHub repo.
You must read and respect that file alongside this brief.
If this brief and the repo’s `agent.md` differ in implementation detail, do not ignore either one — use this brief as product/architecture direction and `agent.md` as repo-specific execution guidance.

This app is not a generic productivity app and not a simple CRUD dashboard.

It is a mobile-first brewery operating system.

Its job is to turn an already-working real brewery operating system into coherent software without losing what makes it operationally valuable.

The app must stay action-first, calm, low-friction, and operationally clear.

---

1. CORE SYSTEM SPLIT

---

Use this system split:

* Airtable = operational facts layer
* Notion = semantic / rules / readiness / translation layer
* App = UI + orchestration + computed behavior

Do not collapse those roles into one system.

Airtable is for real operational data:

* brewery profile
* packaging formats
* ingredients
* receipts
* batches
* brew logs
* lots
* pending movements
* inventory movements
* sales
* declarations

Notion is for semantic structure:

* product foundations
* translation system
* translation starter keys
* system readiness
* execution readiness
* production planning / bottling readiness logic

Do not use Notion as the main operational database.
Do not use Airtable as the semantic rules source.

---

2. PRODUCT INTENT

---

This app should feel:

* smooth like iPhone UI
* calm, clean, and low-friction
* industrial enough for production work
* functional without looking harsh

Visual principles:

* mobile-first layouts
* large tap targets
* rounded cards
* generous spacing
* minimal borders
* soft transitions
* neutral light base
* state colors used sparingly and clearly

State color intent:

* Green = OK
* Orange = Needs attention
* Red = Blocked
* Blue = Action / navigation

The app must never feel like noisy enterprise software.

---

3. APP PRINCIPLE

---

Core operating principle:

User acts → system suggests → user confirms → system structures, links, reminds, and learns

This means:

* the user must always be able to continue work
* the system should guide, explain, and structure
* the system should not silently invent fake data
* missing data should create follow-up actions, lower confidence, or pending states
* missing data should not hard-break normal operation unless absolutely necessary

---

4. PRIMARY UX MODEL

---

The app is action-first.

Primary top-level surfaces:

* Operations = main dashboard / command surface
* Let’s Brew = primary action launcher / operational entry point

Do not treat all modules as equal tabs.

Current direction:

* Operations is the command page
* major modules are previewed there
* each preview can open into a deeper view
* Let’s Brew is the main action trigger

Navigation direction:

* do not rebuild a crowded tab bar
* keep shell ownership centralized
* keep safe-area handling centralized
* keep behavior future-native-safe for eventual iOS wrapping

Back buttons must exist on deeper operational screens so users are never trapped.

---

5. LAUNCH / LANGUAGE SYSTEM

---

This is already defined in Notion and should be implemented exactly as a system.

First launch behavior:

* app opens to language selection screen if no stored language exists
* one-tap selection
* no confirmation button
* no flag icons
* smooth transition into app
* language remembered locally
* future launches skip the language screen
* language remains changeable later in Settings

Supported launch languages:

* Français
* Nederlands
* English
* Deutsch

Settings must include:

* Language
* ability to change it any time
* immediate UI refresh after change

Translation rules:

* use translation keys
* do not hardcode per screen
* if key missing, fall back to English
* missing keys should be easy to identify later

Translation copy style:

* short
* action-first
* operationally clear
* avoid decorative text

---

6. DOMAIN MODEL

---

Use the canonical chain:

Ingredient → Receipt → Batch → Lot → Movement → Sale → Declaration

Core entities:

* BreweryProfile
* PackagingFormat
* Ingredient
* Receipt
* Recipe
* Batch
* BrewLog
* Lot
* PendingMovement
* InventoryMovement
* Sale
* Declaration

The app should show these as human operational objects, not raw Airtable records.

Never expose Airtable IDs in the UI.

---

7. GENERIC LAYER VS BREWERY-SPECIFIC LAYER

---

The system must preserve the split between reusable template logic and live brewery instance data.

Required split:

* Generic Airtable = reusable operational template
* Brewery Airtable = live brewery data
* Generic Notion = reusable semantic template
* Brewery Notion = brewery-specific semantic specialization

Do not architect the app in a way that assumes only one hardcoded brewery forever.

The current brewery-specific live data is the real reference case and later real live app data for this user account.

---

8. WHAT OPERATIONS PAGE SHOULD DO

---

Operations is the main intake surface.

It should:

* surface active batches
* surface pending completions
* surface next actions
* preview tasks
* preview agenda
* preview finances
* preview operational alerts
* allow quick “see more” paths into deeper modules

It should feel like:
“What should I do next?”

It should not feel like:
“Choose one of many equal modules.”

Do not overcrowd it.
Use previews, not giant dashboards.

---

9. WHAT LET’S BREW SHOULD DO

---

Let’s Brew is broader than brewing.
It is the main operational action launcher.

Current intended behavior:

* open from anywhere
* if active batch exists, allow resume
* if no active batch exists, allow select existing batch or create new one
* if selected batch has linked recipe, use recipe as suggestion/prefill only
* if no recipe, allow create, upload, or continue without one

Let’s Brew should also scaffold or route into:

* start/resume brew execution
* quick observation logging
* task continuation
* inventory movement placeholder
* sale placeholder
* compliance placeholder
* documents / exports placeholder
* lightweight future AI intent routing placeholder

Important:
Let’s Brew must feel brewery-specific and action-first, not generic productivity software.

---

10. RECIPE VS EXECUTION RULE

---

Recipe data = plan
Brew-log data = execution reality

Keep them separate.

The app may:

* prefill from recipe
* suggest from recipe
* compare recipe vs actual execution

The app must not:

* silently create fake execution records from the recipe
* confuse plan with reality

---

11. READINESS MODEL

---

Readiness is a major semantic layer.

There are six intended readiness drivers:

* material availability
* process completion state
* system readiness
* measurement reliability
* output lot integrity
* control execution validity

App behavior:

* compute readiness from semantic context when available
* allow fallback behavior when semantic layer is not ready
* explain blocked or reduced-readiness states clearly
* allow override when role permits
* require justification on override
* log overrides

Readiness should not become a dead-end blocker.
It should guide and explain.

---

12. PENDING / INCOMPLETE DATA PHILOSOPHY

---

Never fake precision.

If final data is missing:

* create pending movement
* create incomplete context
* create a follow-up action
* lower certainty if needed
* allow continuation honestly

Do not:

* invent final ledger data
* silently complete inventory facts
* silently convert missing operational information into “done”

---

13. ROLE / PERMISSION DIRECTION

---

Current role direction:

* Owner
* Brewmaster / finance / admin
* Brewer
* Brewer’s assistant
* Viewer

Role logic should affect:

* authority
* visibility
* override ability
* action availability

But do not implement permissions in a way that destroys flow.
The goal is guided operations, not bureaucratic lockout.

---

14. BACKEND ARCHITECTURE TO BUILD NOW

---

Move the app from static frontend + Netlify function dependency into a small Node / Express backend.

Backend must:

* run on port 5000
* serve frontend
* expose /airtable proxy
* expose /notion proxy

This is required because the current static Replit setup cannot execute Netlify serverless functions.

Backend purpose for now:

* proxy external APIs
* centralize credentials
* create a thin contract layer
* prepare for later orchestration logic

Do not overbuild backend at this stage.

No need yet for:

* auth
* own database
* caching
* heavy service layers
* job queues

---

15. AIRTABLE INTEGRATION RULES

---

Airtable is the operational facts source.

Use it for:

* batches
* ingredients
* receipts
* lots
* inventory movements
* tasks if relevant
* live operational records

Requirements:

* keep the Airtable proxy working in Replit
* use secrets
* default base can use AIRTABLE_BASE_ID
* do not expose credentials client-side
* normalize records before surfacing them in UI

UI rule:
never show raw Airtable field/ID mess directly to the user.
The app consumes domain objects, not raw Airtable shapes.

---

16. NOTION INTEGRATION RULES

---

Notion schema already exists conceptually and should be used, not reinvented.

Visible Notion structure already includes:

* Product Foundations
* Translation System
* Translation Starter Keys
* System Readiness
* Execution Readiness
* Production Planning / Bottling Readiness logic

Use Notion as read-oriented semantic source.

Backend should:

* retrieve Notion content
* normalize it into app-friendly semantic objects

Frontend should consume normalized shapes such as:

* translation bundles
* readiness rules
* state definitions
* logic descriptors
* planning dependency descriptors

Do not hardwire the UI directly to raw Notion response structures.

Build a thin normalization layer.

---

17. CLIENT LAYER TO BUILD

---

Create or preserve small client modules such as:

* js/airtable.js
* js/notion.js

These should be simple fetch wrappers against:

* /airtable
* /notion

Frontend code should call helpers, not raw fetch everywhere.

UI rules:

23. FINAL IMPLEMENTATION ATTITUDE

This project should be treated as:
