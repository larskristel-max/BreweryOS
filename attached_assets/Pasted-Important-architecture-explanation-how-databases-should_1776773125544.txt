Important architecture explanation: how databases should work when a new brewery signs up

This app must support more than one brewery.

So the data architecture cannot assume:

* one hardcoded Airtable base forever
* one hardcoded Notion workspace forever
* one shared live dataset for all breweries

The system must be multi-tenant in structure, even if the first live case is only one brewery.

---

1. CORE PRINCIPLE

---

When a new brewery signs up, they should not be dropped into the current brewery’s live data.

They should get their own brewery-specific live layer.

The architecture must preserve a split between:

1. Generic template layer
2. Brewery-specific live layer

This split must exist in both:

* Airtable
* Notion

---

2. WHAT THE GENERIC LAYER IS

---

The generic layer is the reusable system template.

It contains:

* table structure
* field structure
* starter config logic
* canonical object model
* semantic rules
* translation system
* readiness system
* generic product behavior

It does NOT contain the live operational data of a real brewery.

Think of it as:
“the operating system template”

---

3. WHAT THE BREWERY-SPECIFIC LAYER IS

---

The brewery-specific layer is the real live instance for one brewery.

It contains:

* brewery profile
* real ingredients
* real batches
* real lots
* real movements
* real declarations
* real tasks
* brewery-specific semantic adaptations if needed

Think of it as:
“that brewery’s actual app data”

---

4. WHAT SHOULD HAPPEN WHEN A NEW BREWERY SIGNS UP

---

At signup / onboarding, the system should initialize a new brewery instance from the generic template.

That means:

Airtable side:

* create or assign a brewery-specific operational base / dataset
* populate required starter records
* create the brewery profile
* create starter packaging formats
* keep that brewery’s live records separate from all others

Notion side:

* create or assign a brewery-specific semantic workspace / section / dataset cloned from the generic semantic template
* include:

  * state definitions
  * translation rules
  * readiness rules
  * generic product logic
* allow later brewery-specific semantic adaptation without polluting the generic template or another brewery’s logic

---

5. TWO POSSIBLE IMPLEMENTATION MODELS

---

There are two valid patterns.

Pattern A — separate bases / separate semantic spaces per brewery

* each brewery gets its own Airtable base
* each brewery gets its own Notion semantic space/template clone

Pros:

* strongest separation
* easiest mental model
* safer against cross-brew contamination
* easiest for future export / handoff / ownership clarity

Cons:

* more provisioning logic
* more base/workspace management

Pattern B — shared base with tenant separation

* one Airtable base
* one semantic source
* every record carries breweryId / tenantId
* all queries filtered by tenant

Pros:

* easier provisioning
* fewer bases to manage

Cons:

* higher risk of tenant leakage
* more query/filter discipline required everywhere
* more dangerous for operational traceability mistakes
* less clean for cloning and long-term instance ownership

For this product, Pattern A is preferred conceptually.
If Pattern B is used temporarily, tenant separation must be extremely strict.

---

6. PREFERRED PRODUCT DIRECTION

---

Preferred direction:

* Generic Airtable = reusable app template
* Brewery Airtable = live brewery instance
* Generic Notion = reusable semantic template
* Brewery Notion = live brewery semantic specialization

The app should be built so onboarding can initialize a brewery-specific layer from the generic layer.

Do not hardcode the current brewery as if it were the only future customer.

---

7. WHAT THE APP SHOULD STORE FOR TENANCY

---

The app needs a concept of the active brewery context.

At minimum, keep a runtime context like:

* breweryId
* breweryName
* operational source reference
* semantic source reference
* language
* timezone
* country
* excise enabled yes/no

This runtime brewery context must be initialized before normal app loading.

The frontend should not guess which source to use.
The app context must tell it which operational and semantic sources belong to the active brewery.

---

8. WHAT ONBOARDING SHOULD DO

---

When a new brewery signs up, onboarding should do something like this:

Step 1:
Create brewery account / brewery identity

Step 2:
Initialize brewery operational layer from generic Airtable template

* create Brewery Profile
* create starter Packaging Formats
* create starter config
* prepare empty operational tables/records

Step 3:
Initialize brewery semantic layer from generic Notion template

* translation baseline
* readiness baseline
* state definitions
* semantic rules baseline

Step 4:
Store the source mapping for that brewery
Example:

* this brewery uses Airtable source X
* this brewery uses Notion source Y

Step 5:
Enter the app in a real initialized state
Not a demo state

---

9. HOW THE BACKEND SHOULD THINK ABOUT IT

---

The backend should not think:
“there is one Airtable base and one Notion setup”

The backend should think:
“for the currently active brewery, resolve the correct operational source and semantic source”

So internally there should be a source resolution layer.

Pseudo-logic:

* get active brewery context
* resolve Airtable base for that brewery
* resolve Notion semantic source for that brewery
* use those sources for all reads/writes

This is important because later the app may support:

* multiple breweries
* template cloning
* demo environments
* staging instances
* native app accounts
* per-brewery exports

---

10. WHAT MUST NEVER HAPPEN

---

Never let:

* brewery A see brewery B’s live data
* brewery-specific semantic tweaks overwrite the generic template
* one brewery’s readiness logic silently affect another brewery
* one brewery’s batches/lots/inventory live in a shared unscoped pool by accident

Cross-brew contamination would be a critical product failure.

---

11. SHORT PRACTICAL RULE FOR CURRENT BUILD

---

Even if the current build still uses one live brewery only, build the code as if source resolution already exists.

Meaning:

* do not hardcode a single forever-base assumption into core app logic
* put source resolution behind config/context
* normalize data around breweryId / active brewery context
* keep onboarding architecture compatible with future instance creation

---

12. WHAT TO IMPLEMENT NOW VS LATER

---

Implement now:

* brewery context concept
* source resolution concept
* no hardcoded forever-single-brewery architecture
* onboarding response shape that can support initialization

Can be simplified for now:

* actual automated cloning/provisioning of new Airtable/Notion instances
* advanced multi-tenant account management UI
* billing / subscription management
* per-brewery admin console

Meaning:
Build the architecture now so those can be added later without rebuild.

---

13. SIMPLE MENTAL MODEL

---

Generic layer = app template
Brewery layer = that brewery’s real operating system instance

New brewery signup should create:
template → live brewery instance

not:
template → shared global mess

---

14. FINAL IMPLEMENTATION RULE

---

Treat brewery onboarding as:
“instantiate a clean brewery-specific operational layer and semantic layer from reusable templates”

not:
“add another brewery into the same undifferentiated data bucket”

This distinction is fundamental to the future of the app.
