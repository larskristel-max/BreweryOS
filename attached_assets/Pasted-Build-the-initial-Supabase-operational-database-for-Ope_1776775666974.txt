Core architecture:

* Supabase/Postgres = operational database
* Notion = semantic / readiness / translation layer
* Frontend = TypeScript + Tailwind
* Backend/service layer = normalization, business logic, privileged writes

Goal:
Create a clean Supabase schema that faithfully translates the current Airtable operational structure into a normalized PostgreSQL design suitable for long-term use.

---

1. GENERAL DATABASE RULES

---

1. Use PostgreSQL in Supabase.

2. This is multi-brewery-capable architecture.
   Every operational table must support tenancy from the beginning.

3. Prefer a shared database with strict brewery separation by `brewery_id`.

4. Every major operational table must include:

* `id uuid primary key default gen_random_uuid()`
* `brewery_id uuid not null`
* `created_at timestamptz default now()`
* `updated_at timestamptz default now()` where appropriate

5. Do not use Airtable IDs as primary identifiers in the new system.

6. Use proper foreign keys.

7. Keep schema clean, typed, and extensible.
   Do not overbuild.

5. IMPORTANT PRODUCT RULES TO ENFORCE IN THE SCHEMA

---

1. Recipe data is the plan.
2. Brew-log and execution tables are actual reality.
3. Do not merge planned and actual into one table.
4. Pending movements must exist as a first-class concept.
5. Inventory movements must be the authoritative stock ledger.
6. Lot genealogy must be possible through parent/source/destination relationships.
7. Brewery isolation must be possible through brewery_id on all major operational tables.
8. The schema must support future row-level security by brewery and user role.

---

6. RELATIONSHIP / NORMALIZATION RULES

---

1. Replace Airtable multiple-record-links with proper relational foreign keys and junction tables only where needed.
2. Avoid Airtable-style reverse-link duplication.
3. Remove duplicated reverse-link artifacts from the new schema.
4. Use normalized one-to-many / many-to-one relations where possible.
5. Only use junction tables where a true many-to-many relationship is required.

---

7. WHAT NOT TO DO

---

Do NOT:

* invent startup/SaaS-generic tables unrelated to brewery operations
* flatten everything into one giant batch table
* collapse recipe and execution into the same object
* remove pending movements
* use raw Airtable IDs
* keep Airtable-specific field naming when a cleaner SQL name exists
* hardcode one-brewery-forever assumptions
* add frontend code at this step
* add auth/billing flows at this step
* build full RLS policies yet unless needed minimally
